| 层级 | 主要职责 | 关键模块 | 设计原则 |

### 1.3 层级交互原则

```
UI层 → Store层 → Engine层 → Utils层 → Data层
 │        │         │          │         │
 │        │         │          │         └─ 只读访问
 │        │         │          └─ 纯函数调用
 │        │         └─ 状态读写、方法调用
 │        └─ 状态同步、动作派发
 └─ 事件监听、状态订阅
```

**关键约束**：
- 下层不能依赖上层（Engine 不能 import Store 或 UI 组件）
- 同层模块间可互相调用，但应保持最小依赖
- 状态变更只能通过 Engine 内部方法或 Store 的 action
- 跨层通信通过回调函数（`onStateChange`, `onChestOpened`, `onShopOpen`）

## 2. 完整调用链路图

### 2.1 游戏启动链路

```
用户点击"开始游戏"
    │
    ▼
[MainMenu.tsx] 调用 engine.startGameWithClass()
    │
    ├─ 加载玩家存档 (storage.ts)
    ├─ 计算天赋效果 (talents.ts)
    ├─ 生成初始符文 (runes.ts)
    ├─ 生成地牢 (dungeon.ts)
    ├─ 生成怪物 (monsters.ts)
    ├─ 生成宝箱 (dungeon.ts)
    ├─ 初始化玩家状态
    ├─ 初始化宠物 (pets.ts)
    ├─ 计算技能 (runes.ts)
    ├─ 播放BGM (AudioManager.ts)
    └─ 调用 notifyStateChange()
            │
            ▼
    [GameEngine] onStateChange 回调
            │
            ▼
    [App.tsx] updateFromEngine()
            │
            ▼
    [gameStore.ts] 更新 store 状态
            │
            ▼
    React 组件 re-render (GameCanvas, GameHUD, etc.)
```

```

            │
```

## 6. 总结

### 6.1 架构优点

1. **分层清晰**：UI层 → Store层 → Engine层 → Utils层 → Data层的划分基本合理
2. **类型安全**：完整的TypeScript类型定义，数据结构清晰
3. **数据驱动**：游戏内容（符文、怪物、装备等）通过配置数据定义
4. **状态隔离**：游戏状态集中在GameEngine，UI状态通过Store同步
5. **回调机制**：Engine通过回调与UI层通信，避免反向依赖

### 6.2 核心问题

1. **GameEngine 成为上帝类**：承担了过多职责，违反单一职责原则
2. **违反开闭原则**：AI类型、技能类型、宠物技能等硬编码
3. **逻辑混杂**：渲染与逻辑混杂、输入与业务混杂
4. **方法过大**：多个方法超过100行甚至200行
5. **代码重复**：特效播放、存档调用等重复代码较多

### 6.3 架构建议优先级

| 优先级 | 建议 | 预计收益 |
|-------|------|---------|
| P0 | 拆分 GameEngine，抽离 RenderSystem | 可维护性↑50%，可测试性↑40% |
| P1 | 抽离 MonsterAISystem，使用策略模式 | 可扩展性↑60%，代码复用↑30% |
| P1 | 抽离 StatusEffectSystem | 可测试性↑40%，Bug率↓20% |
| P2 | 抽离 SkillSystem，使用策略模式 | 可扩展性↑50% |
| P2 | 抽离 PetSystem | 代码组织↑30% |
| P2 | 抽离 PotionSystem | 代码组织↑30% |
| P3 | 抽离 InputSystem | 可扩展性↑20% |
| P3 | 抽离 ParticleSystem | 性能优化空间↑30% |
